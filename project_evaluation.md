# CarMoneyPit 프로젝트 평가 보고서

## 1. 요약 (Executive Summary)
**현재 상태:** 로직 및 UX 프로토타입 (우수함)
**SEO/수익화 준비 상태:** ❌ **치명적 결함 (Critical Failure)**

현재 구현은 **로직(후회 이론)**과 **UX(2단계 흐름)** 측면에서 기획 의도를 완벽하게 따르고 있습니다. "계산기"로서는 훌륭하게 작동합니다.

**하지만**, 현재 상태로는 구글 검색 트래픽이 **"0"**에 수렴할 것입니다.
사용자의 목표는 "검색 결과 상위 선점(SERP Dominance)"이지만, 현재 아키텍처는 결과 페이지를 검색 엔진에게 **숨기고** 있습니다.

---

## 2. 기술적 분석

### ✅ 잘된 점
1.  **후회 로직 코어 (`RegretCalculator.java`)**
    -   `RF(수리 후회) vs RM(정리 후회)` 비교 로직이 기획안대로 엄격하게 구현되었습니다.
    -   "미래 고장 위험" 및 "유동성 페널티" 로직이 `계산식.md` 스펙과 일치합니다.
2.  **UX 흐름 (`index.jte` -> `result.jte`)**
    -   3개의 입력만 받는 빠른 판정 단계가 깔끔합니다.
    -   HTMX 기반의 시뮬레이션 랩(Simulation Lab)은 페이지 새로고침 없이 역동적인 느낌을 줍니다.

### 🚨 치명적 문제점 (수정 필수)

#### 1. "POST" 방식의 함정 (SEO 킬러)
현재 `CarDecisionController.java`는 요청을 `POST` 방식으로 처리합니다:
```java
@PostMapping("/analyze") // <--- 구글봇은 '제출' 버튼을 누르지 않습니다.
public String analyze(...) { ... }
```
-   **문제:** 결과 페이지는 사용자가 폼을 제출한 "후"에만 존재합니다. 특정 상황(예: 15만 마일 세단)에 대한 고유 URL이 없습니다.
-   **결과:** 구글은 "판정 결과" 페이지를 크롤링할 수 없습니다. 색인 가능한 페이지는 오직 1개(메인 페이지) 뿐입니다.
-   **영향:** *"15만 마일 뛴 2012년식 혼다 어코드 고쳐야 하나요?"* 같은 검색어를 타겟팅할 수 없습니다. 웹상에 해당 페이지가 존재하지 않기 때문입니다.

#### 2. 콘텐츠 볼륨 부족
-   상위권을 장악하려면 수천 개의 랜딩 페이지가 필요합니다 (프로그래매틱 SEO).
-   현재 앱은 사용자의 입력을 수동적으로 기다립니다. 인기 있는 차종/주행거리 조합에 대해 페이지를 **미리 생성**하거나 **동적 URL**로 제공해야 합니다.

## 3. 제안: 프로그래매틱 SEO (pSEO)로의 전환

"계산기"에서 "콘텐츠 엔진"으로 중심축을 옮겨야 합니다.

### 아키텍처 변경
**변경 전:**
`사용자 입력 -> POST /analyze -> 결과 HTML`

**변경 후:**
`GET /verdict/{vehicleType}/{mileage} -> 결과 HTML`

**생성될 URL 예시:**
-   `lifeverdict.com/verdict/sedan/150000-miles`
-   `lifeverdict.com/verdict/suv/broken-transmission`

### 실행 계획 (Action Plan)
1.  **컨트롤러 리팩토링:** `POST /analyze` 방식을 `GET /verdict/...` 경로 변수 방식으로 변경합니다.
2.  **사이트맵 생성:** 일반적인 시나리오(예: "고주행 세단", "SUV 변속기 고장")를 커버하는 사이트맵을 생성하여 구글에 제출합니다.
3.  **내부 링크 연결:** 메인 페이지에 "인기 시나리오" 링크를 추가하여 구글봇이 깊숙한 판정 페이지까지 찾아갈 수 있게 길을 터줍니다.

비즈니스 목표 달성을 위해, 즉시 **로직을 URL 경로로 노출**하는 작업이 필요합니다.
